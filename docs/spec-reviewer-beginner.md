# Antigravity 專案規格書終極新手審查 (spec-reviewer-beginner)

# 功能說明

扮演多合一專家團隊，將艱澀的軟體專案規格書轉化為「完全外行也能聽懂」的深度分析報告，並確保專案邏輯無誤。

# 解決什麼問題

解決非技術背景人員（老闆、PM、初學開發者）看不懂專業規格書的痛點，並預防因資訊不對稱、技術術語堆砌而導致的溝通誤解、開發失敗或預算浪費。

# 適合什麼情境

1. **專案啟動前審核**：在投入資源前，確認規格書是否有邏輯漏洞。
2. **非技術溝通橋樑**：老闆或商業負責人需要了解技術細節，但聽不懂專業術語。
3. **初學者學習導引**：新手開發者接手專案，需要有人把架構「翻譯」成白話文。
4. **可行性快速評估**：確認 Antigravity 平台是否能實作規格書中的功能。
5. **預算與風險控管**：評估專案難度與開發成本是否合理。

# 如何觸發

請直接提供專案規格書內容，或輸入以下指令：

* **範例1**：「請幫我審查這份專案規格書：[貼上規格書內容]」
* **範例2**：「我是一個剛創業的老闆，請用白話文告訴我這份技術文件到底在做什麼？」
* **範例3**：「分析這份 Antigravity 規格書，列出所有缺少的細節並評估開發難度。」
* **範例4**：「請扮演專家團隊，審查這份專案的 AI Agent 流程是否合理？」
* **範例5**：「幫我把這份規格書轉化為一頁式老闆懶人包，並檢查有哪些做不出來的風險。」

# 輸入內容

* **專案規格書全文**：包含功能說明、業務流程、技術架構、API 定義或資料表設計。
* **特定審查需求**（選填）：例如特別強調資安、商業價值或學習難度。

# 輸出內容

將嚴格執行 10 個階段的深度分析報告：
1. **缺失補件提醒**：若規格書不完整，先列出需補齊的項目。
2. **三維度白話翻譯**：用國中生、上班族、老闆角度解釋專案，並提供技術術語的「生活化對照表」。
3. **功能與操作拆解**：將技術功能轉化為具體的使用者價值與操作步奏。
4. **商業與可行性分析**：ROI 評估、Antigravity 原生功能配對與開發難度量表。
5. **漏洞與風險診斷**：找出缺失的例外處理、通知機制或資安風險。
6. **新手學習地圖**：提供 30/60/90 天的技術學習路線圖。
7. **可視化圖表**：輸出 Mermaid 格式的系統架構圖與資料流圖。
8. **MVP 路線圖**：分階段建議最優先開發的功能。
9. **一頁式老闆懶人包**：最後提供決策核心重點。

# 注意事項

* **資訊完整度**：若提供的規格書過於簡略（例如只有一句話），Skill 會要求使用者補充細節以維持分析品質。
* **AI 限制**：分析結果基於當前規格書，AI 可能存在幻覺，重大決策仍需與資深架構師複核。
* **新手導向**：本 Skill 會刻意避開單純的技術名詞堆砌，若需要極度專業的工程對話，請明確告知。

# 常見問題

* **Q: 為什麼要用生活比喻解釋 API 或 Docker？**
  * A: 為了確保沒有任何技術背景的決策者也能完全理解系統背後的運作邏輯。
* **Q: 這個 Skill 可以幫我寫程式碼嗎？**
  * A: 它的核心是「審查」與「分析」，雖然能評估代碼邏輯，但主要目標是優化規格書與溝通流程。
* **Q: 如果規格書寫得很爛會怎樣？**
  * A: Skill 會在第一階段直接「打槍」並列出清單，要求補齊後才開始深度分析，這能節省無謂的溝通時間。
