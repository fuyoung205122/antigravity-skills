# Doc Co-Authoring Workflow (文件協同寫作工作流)

# 功能說明

引導使用者透過「脈絡收集、結構提煉、讀者測試」三階段結構化流程，高效且高品質地協同創作各式專業文檔。

# 解決什麼問題

解決撰寫重要文檔時常遇到的痛點：作者與 AI 資訊不對稱（缺乏脈絡）、文檔結構混亂、充斥無意義的 AI 罐頭文字（slop），以及因「知識詛咒」導致寫出他人看不懂、有閱讀盲點的文檔。

# 適合什麼情境

* **產品與技術規劃**：撰寫產品需求文檔 (PRD)、技術規格書 (Technical Specs)、系統設計文檔 (Design Doc)。
* **架構與決策紀錄**：起草架構決策紀錄 (ADR) 或徵求意見稿 (RFC)。
* **商業與專案提案**：撰寫專案提案 (Proposal)、商業分析報告。
* **團隊規範與文檔重構**：重新整理或優化既有的團隊協作指南與知識庫文章。

# 如何觸發

當您提及需要寫作、起草或規劃任何實質性文檔時即可觸發。

範例1：
> 「我想寫一份關於新功能上線的 PRD，但不知道從何開始。」

範例2：
> 「幫我起草一份技術規格書（Tech Spec），我們要將資料庫遷移到 PostgreSQL。」

範例3：
> 「我需要寫一封給高層的專案提案（Proposal），說服他們投資 AI 客服系統。」

範例4：
> 「我們團隊需要一份架構決策紀錄（ADR）來規範 API 的設計標準。」

範例5：
> 「我想寫一個關於如何優化系統效能的 RFC（Request for Comments）。」

# 輸入內容

1. **基本設定**：文檔類型、目標受眾、預期影響、特定模板或格式限制。
2. **原始脈絡（Info Dump）**：項目背景、團隊討論紀錄、不採納替代方案的原因、時間壓力、技術依賴性或利益相關者的顧慮（可隨意無結構地提供，或透過整合工具拉取）。
3. **段落偏好回饋**：在腦力激盪與草擬階段，針對各章節內容提供「保留/刪除/合併」的具體意見。

# 輸出內容

1. **結構化的文檔骨架**：包含所有必要章節與占位符（Placeholder）的 Markdown 檔案或 Artifact。
2. **逐章精雕細琢的內文**：經過反覆迭代、修飾，去蕪存菁的高質量文本。
3. **讀者測試報告**：經由「虛擬讀者（無脈絡的 Claude）」測試後，找出的閱讀盲點、含糊之處與矛盾點，並完成修正的最終版文檔。

# 注意事項

* **非一次性生成**：這是一個「協同（Co-authoring）」工作流，需要您在各階段積極配合互動與提供反饋，無法一鍵直接產出終版。
* **修改建議方式**：在修改草稿時，建議您告訴 AI「如何修改」（例如：*請把第三段寫得更精簡*），而非直接動手改 Artifact，這能幫助 AI 快速學習並掌握您的寫作風格。
* **讀者測試限制**：若您使用的是無 Sub-agent 權限的環境（如 Claude.ai 網頁版），需要您配合手動開啟新對話視窗來進行「讀者測試」步驟。

# 常見問題

**Q：我可以跳過這三個階段，直接叫你幫我生出整篇文檔嗎？**
A：可以。如果您拒絕此工作流，AI 將退回一般的「自由寫作模式」。但強烈建議使用此流程，這能確保產出的文檔真正符合您的需求，且對真實讀者而言好讀易懂。

**Q：什麼是「讀者測試（Reader Testing）」？**
A：這是本工作流的特色。在文檔發佈前，由一個完全沒有本對話脈絡的「全新 Claude 實例」來試讀您的文檔，並回答讀者可能會問的問題。這能有效抓出作者因過於熟悉背景而忽略的「資訊斷層（Blind spots）」。

**Q：為什麼在修改時，AI 都是局部替換（str_replace）而不是整篇重寫？**
A：局部精確修改可以節省 Token，並避免在重新生成整篇文檔時，意外遺失或改動了先前已經確認好的優良段落。
